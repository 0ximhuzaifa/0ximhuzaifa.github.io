Summary
#

Authority is a medium-difficulty Active Directory machine that demonstrates how multiple seemingly minor weaknesses can be chained into a complete domain compromise. The attack begins with an anonymously accessible SMB share containing Ansible deployment files, leading to the recovery of encrypted Ansible Vault secrets. After cracking the vault password, valid credentials are obtained for the PWM Password Self Service application. A misconfigured LDAP testing feature is then abused to capture LDAP proxy credentials in cleartext, providing an initial foothold on the domain. Finally, the compromise is escalated by exploiting a vulnerable Active Directory Certificate Services (AD CS) certificate template (ESC1) together with the default MachineAccountQuota setting, resulting in full administrative control over the domain.

Initial Access
#

Web Application Enumeration
#

We start by enumerating the HTTP applications at ports 80 and 8443. Port 80 shows a default IIS server landing page.

As the IIS page did not reveal any useful functionality, directory brute forcing was performed to identify hidden applications or administrative interfaces, but nothing useful was found:

└─$ ffuf -u 'http://10.129.11.43/FUZZ' -ic -w /usr/share/wordlists/seclists/Discovery/Web-Content/raft-medium-directories.txt -t 50 -fc 404

Moving to the https application on port 8443, the website redirects us to a login page of PWM Password Self Service.

Attempting common default credentials such as admin:admin was unsuccessful and showed an error notice. The error showed that when the application tried to connect to the configured LDAP url “ldaps://authority.authority.htb:636” as the proxy user “svc_ldap”, the connection failed.

Checking the two buttons below the Sign In option redirects to another login page that is just asking for a password and no username. From the same page, previous authentication table data revealed a username “svc_pwn” in its Distinguished Name (DN) format as “CN=svc_pwm,CN=Users,DC=htb,DC=corp (default)”.

So far there is not much for us to enumerate further in the web application.

Cracking Ansible Vault Secrets
#

Ansible is an automation tool that uses playbooks to configure systems. Finding these playbooks implies configuration data or deployment secrets might be exposed in plaintext or encrypted vaults.

Since the downloaded files contain several Ansible playbooks, we begin reviewing each role individually, looking for credentials, deployment scripts, or configuration files containing sensitive information. We find some tomcat and winrm plaintext credentials but neither of them worked for the domain or the PWM login pane. Ansible roles commonly store configuration values in defaults/main.yml. Discovering encrypted Ansible Vault entries suggests that sensitive credentials are being deployed automatically and may be recoverable if the vault password can be cracked.

└─$ ansible-vault decrypt encryptedVault-1

└─$ ansible-vault decrypt encryptedVault-2

The first two values correspond to the PWM administrator username and password but the third one is of no use as it didn’t work for svc_ldap’s password.

Shell as svc_ldap
#

After login, we see a screen showing configurations for the PWM application. We are also given the option to download its main configuration file “PwmConfiguration.xml” from the “Download Configuration” button.

The file has the encrypted credentials of svc_ldap user but we can’t decrypt it. Looking around in the other panel (Configuration Editor), we find a setting where the LDAP URL of the server is configured. The same page has the svc_ldap in the LDAP Proxy User section but again the password is hidden and not shown.

PWM validates LDAP connectivity by attempting to bind to the configured LDAP server using the configured proxy account. By replacing the LDAP server address with an attacker-controlled host, we can observe the application’s LDAP bind request and capture the credentials transmitted during the authentication attempt.

We can start netcat listener and add our tun0 interface IP and set it as the LDAP URL for the application to try checking if the application sends us the cleartext credentials of svc_ldap. Using the Add Value button in the LDAP URLs section:

Start the netcat listener on the port 389:

nc -lvnp 389

Now click the “Test LDAP Profile” button on the top of the page and see in the netcat listener back. We have successfully received the cleartext credentials of svc_ldap

Using the provided credentials, we can login to the host using evil_winrm:

└─$ evil-winrm -i 10.129.11.43 -u svc_ldap -p 'lDaP_1n_th3_cle4r!'

The user.txt can be found in the desktop folder of svc_ldap user.

The tool finds that the CorpVPN template is vulnerable to ESC1. ESC1 allows an attacker to request a certificate containing an arbitrary SAN. If the attacker can enroll for the vulnerable template, they can obtain a certificate impersonating another user, including Domain Administrators.

The exploitation of this template has following requirements:

  • Low privilege users can enroll
  • Manager approval disabled
  • No authorized signatures required
  • Client Authentication enabled
  • Enrollee supplies subject enabled

Almost all the requirements are ok except the 1st one which requires a user that can ask for the template’s certificate. Looking in the certipy output shows that Domain Computers have the rights to enroll and by default in Active Directory, all domain users are allowed to create and join 10 machine accounts to the domain. This property is determined by the value of MachineAccountQuota of each domain user and we can check if svc_ldap can create machine accounts or not with Netexec:

└─$ nxc ldap 10.129.11.43 -u 'svc_ldap' -p 'lDaP_1n_th3_cle4r!' -M maq

Machine Account Creation
#

The output confirms that the default MachineAccountQuota is 10, allowing authenticated users to create up to ten machine accounts. We can add a machine account to the domain with impacket-addcomputer:

└─$ impacket-addcomputer authority.htb/svc_ldap:'lDaP_1n_th3_cle4r!' -computer-name 'Attacker-PC$' -computer-pass 'Password@12345' -dc-ip 10.129.11.43

[*] Successfully added machine account Attacker-PC$ with password Password@12345.

LDAP Shell Access & Group Modification
#

The error “KDC_ERR_PADATA_TYPE_NOSUPP(KDC has no support for padata type)” occurs when the Domain Controller does not support PKINIT. Alternatively, we can use schannel authentication to get a ldap shell:

└─$ certipy auth -pfx administrator.pfx -username Administrator -domain authority.htb -dc-ip 10.129.11.43 -ldap-shell

In the help menu, we have different opions for privilege escalation. We can simply add our user svc_ldap to the Administrators group with:

# add_user_to_group svc_ldap Administrators

Adding user: svc_ldap to group Administrators result: OK

As the user is member of Administrators group now, we can again authenticate as svc_ldap using evil-wirnm but this time we will be able to navigate the Administrator’s directory which was not allowed before.

└─$ evil-winrm -i 10.129.11.43 -u svc_ldap -p 'lDaP_1n_th3_cle4r!'

The flag is available in the root.txt on Administrator’s Desktop.
