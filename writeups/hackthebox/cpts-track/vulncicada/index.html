Summary
#

VulnCicada is a Medium-difficulty Windows Active Directory machine that begins with the discovery of a password hidden in an image exposed through an anonymous NFS share. The recovered credentials provide access to domain resources, leading to the identification of an AD CS ESC8 misconfiguration. Since NTLM is disabled, Kerberos relay is achieved by abusing the marshal/unmarshal SPN behavior to bypass self-relay restrictions and obtain a certificate for the Domain Controller machine account. This certificate is then used to authenticate as the DC, perform a DCSync attack, recover the Administrator’s credentials, and fully compromise the domain.

The output shows that there is no vulnerable template but the ADCS Server is itself vulnerable to ESC8. SpecterOPS define ESC8 as:

ESC8 is a misconfiguration in Active Directory Certificate Services (AD CS), which stems from insufficiently protected web enrollment services. Specifically, if a web enrollment service is not hosted over HTTPS with Channel Binding enforced, NTLM authentication attempts in the domain network can be relayed to the web enrollment service to request certificates for other domain principals.

We can’t relay NTLM authentication to the HTTP enrollment endpoint as NTLM is disabled in the domain. Researching for Kerberos authentication relay to ADCS HTTP revealed the article from Synacktiv.

The article describes a technique that abuses the way Windows marshals and unmarshals SPN (Service Principal Name) information during SMB authentication. Normally, when a client authenticates to a service using Kerberos, the ticket is issued for the exact SPN requested by the client. Since the ticket is cryptographically tied to that SPN, relaying it to another service is normally impossible because the target service cannot decrypt a ticket intended for a different SPN.

During SMB authentication, however, Windows uses the CredMarshalTargetInfo API to append a serialized (marshaled) representation of the target information to the hostname. The resulting hostname appears unusual, for example:

DC-JPQ2251UWhRCAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAYBAAAA

Although this looks like a completely different hostname, Windows recognizes the appended blob as marshaled data. Before requesting the Kerberos ticket, it automatically unmarshals the value using the corresponding CredUnmarshalTargetInfo API, recovering the original hostname DC-JPQ225. Consequently, the KDC issues a Kerberos ticket for the legitimate SPN:

cifs/DC-JPQ225.cicada.vl

DNS resolution, however, occurs before the unmarshaling step. Therefore, if an attacker creates a DNS record with the marshaled hostname pointing to their own machine, the coerced authentication is sent to the attacker’s server while still carrying a valid Kerberos ticket for the original domain controller. Since the ticket remains valid for the intended SPN, tools such as krbrelayx can relay it to another service running under the same machine account, such as the AD CS HTTP enrollment endpoint vulnerable to ESC8.

To exploit this behavior, we create a DNS record whose hostname consists of the original hostname followed by the marshaled target information. The record points to our attacking machine instead of the domain controller. When the DC later connects to this hostname, DNS directs the connection to us, while Windows internally unmarshals the hostname back to DC-JPQ225, causing the Kerberos ticket to be generated for the legitimate SPN.

Obtaining the Domain Controller Certificate
#

Looking back in the krbrelayx terminal shows that the connection received successfully and after retrieving the TGS from the request, it gets the certificate and write it to the DC-JPQ225.pfx file

Notice that the relay target is the legitimate AD CS Web Enrollment endpoint (http://dc-jpq225.cicada.vl/certsrv/certfnsh.asp). Although the incoming Kerberos authentication reaches our attacker-controlled server because of the malicious DNS record, krbrelayx extracts the Kerberos ticket and immediately relays it to the genuine HTTP service running on the domain controller. Since both the SMB service and the AD CS Web Enrollment service run under the same machine account, the service can successfully decrypt and accept the relayed Kerberos ticket.
